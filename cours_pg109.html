Bibliothèques Statiques
Pourquoi cette notion ?

Imagine que tu as écrit une fonction ticket_nouveau() dans un fichier ticket_tab.c. Cette fonction est utile dans plusieurs programmes différents. Sans bibliothèque, tu devrais à chaque fois recompiler ticket_tab.c et traîner le fichier .o partout — c'est pénible à maintenir, surtout quand tu as des dizaines de fichiers .o.

L'idée d'une bibliothèque statique, c'est de regrouper un ou plusieurs fichiers objets (.o) dans un seul fichier .a (pour "archive"). Ça simplifie la distribution et la compilation : au lieu de lister plein de .o, tu lies juste avec une bibliothèque.

Comment ça marche — 3 étapes
1
Compiler le .c en .o
Compilation classique sans édition de liens.
$ gcc -c ticket_tab.c -o ticket_tab.o
2
Créer la bibliothèque statique
La commande ar (pour "archive") emballe un ou plusieurs .o dans un fichier .a. Le nom doit respecter la convention lib*.a.
$ ar rcs libticket_tab.a ticket_tab.o
Convention de nommage : lib[nom].a → ex : libticket_tab.a
3
Compiler le programme en liant avec la bibliothèque
$ gcc -o cafeteria_tab cafeteria.c -L. -lticket_tab
optionrôlevaleur icilie avec

Options importantes
-L.
Dit au compilateur de chercher les bibliothèques dans le dossier courant (.). Tu peux remplacer le point par n'importe quel chemin.
-lticket_tab
Dit de lier avec libticket_tab.a. Le préfixe lib et le suffixe .a sont ajoutés automatiquement.
Bibliothèques Dynamiques
Différence avec les statiques

Statique (.a)

Le code de la bibliothèque est copié dans l'exécutable. Il est autonome mais plus gros.

Dynamique (.so)

À la compilation, on met juste un lien. Le code n'est pas copié — il est chargé en mémoire au moment où tu lances le programme.

Pourquoi c'est utile ?

Imagine que 10 programmes différents utilisent tous ticket_nouveau() :

  • Avec une bibliothèque statique : le code de ticket_nouveau() est copié 10 fois, une fois dans chaque exécutable.
  • Avec une bibliothèque dynamique : le fichier libticket_tab.so existe en un seul exemplaire sur le disque, et les 10 programmes vont tous le chercher au moment de l'exécution.
Comment on la crée — 2 étapes
1
Construire la bibliothèque dynamique
$ gcc -shared -fPIC ticket_tab.c -o libticket_tab.so
-shared
Dit à gcc de produire une bibliothèque partagée, pas un exécutable.
-fPIC
"Position Independent Code" — le code machine produit pourra fonctionner peu importe où il est chargé en mémoire.
2
Compiler le programme en liant avec la bibliothèque
Exactement la même commande que pour la statique !
$ gcc -o cafeteria_tab cafeteria.c -L. -lticket_tab
GCC cherche le .so (dynamique) en priorité avant le .a (statique).
La grande différence : à l'exécution

Avec la statique, tu fais ./cafeteria_tab et ça marche direct.

Avec la dynamique, le système doit trouver libticket_tab.so au moment du lancement. Si il ne la trouve pas → erreur. Il faut lui dire où chercher avec LD_LIBRARY_PATH :

$ LD_LIBRARY_PATH=$(pwd) ./cafeteria_tab
Ça dit "cherche les bibliothèques dynamiques aussi dans le dossier courant".
Manipulation de Bibliothèques
Les deux commandes utiles

nm $binaire — liste les symboles d'un binaire (noms des fonctions et variables globales). Ça te permet de vérifier ce que contient un .o, un .a ou un .so. Par exemple :

$ nm libticket_tab.a

Tu verrais ticket_nouveau dans la liste → la fonction est bien dans ta bibliothèque.

ldd $binaire — liste les bibliothèques dynamiques dont un exécutable a besoin pour tourner. Par exemple :

$ ldd cafeteria_tab
Ça te montrerait : libticket_tab.so => /home/etudiant/libs/libticket_tab.so — tu vois direct si le système trouve la bibliothèque ou pas.
Les deux variables d'environnement

Tu connais déjà LD_LIBRARY_PATH (pour l'exécution). Il y en a une deuxième :

  • LIBRARY_PATH → utilisée à la compilation. Remplace l'option -L. Au lieu de faire gcc ... -L/home/etudiant/libs, tu peux faire :
$ export LIBRARY_PATH=/home/etudiant/libs
$ gcc -o cafeteria cafeteria.c -lticket_tab
  • LD_LIBRARY_PATH → utilisée à l'exécution. On en a déjà parlé.
LIBRARY_PATH c'est pour gcc  |  LD_LIBRARY_PATH c'est pour le programme quand il tourne.
Forcer l'utilisation d'une bibliothèque statique

Comme on l'a vu, gcc préfère la dynamique (.so) par défaut. Si tu as les deux versions et que tu veux forcer la statique, deux façons :

$ gcc -o cafeteria cafeteria.c -L. -l:libticket_tab.a

Le -l: avec le nom complet du fichier force gcc à prendre exactement ce fichier. Ou plus simplement, tu donnes directement le fichier :

$ gcc -o cafeteria cafeteria.c libticket_tab.a
Le conseil important
⚠ Toujours mettre -L et -l à la fin de la commande gcc !

Pourquoi ? Parce que gcc traite les arguments dans l'ordre. Il doit d'abord voir les fichiers .c pour savoir quels symboles il lui manque, et ensuite il cherche ces symboles dans les bibliothèques. Si tu mets la bibliothèque avant le .c, gcc ne sait pas encore ce qu'il cherche et il ignore la bibliothèque.

$ gcc -o cafeteria cafeteria.c -L. -lticket_tab ← BIEN
$ gcc -L. -lticket_tab -o cafeteria cafeteria.c ← RISQUE DE NE PAS MARCHER
Fichiers d'en-tête
Résumé

Les fichiers .h contiennent les prototypes des fonctions. Quand tu utilises une bibliothèque, tu n'as pas le .c mais tu as besoin du .h.

Deux vérifications séparées :

  • .h à la compilation → vérifie que tu appelles les fonctions correctement (nombre de paramètres, types). Sans .h, juste un warning: implicit declaration of function, aucune vérification.
  • .so / .a à l'édition de liens → contient le vrai code machine. Le linker vérifie juste que la fonction existe, pas les paramètres.
pkg-config
Le problème

Tu veux compiler avec SDL. Mais quelles options passer à gcc exactement ? Où sont les fichiers .h ? Comment s'appelle le -l ? Tu ne sais pas forcément.

La solution

C'est un programme qui te donne les bonnes options à passer à gcc. Les bibliothèques bien faites fournissent un fichier .pc qui contient toutes ces informations.

Les trois commandes

Lister les bibliothèques disponibles :

$ pkg-config --list-all | grep -i sdl
Ça cherche toutes les bibliothèques installées qui contiennent "sdl" dans le nom.

Obtenir les options de compilation (les -I, pour trouver les .h) :

$ pkg-config --cflags sdl2
Résultat : -I/usr/include/SDL2 -D_REENTRANT — les fichiers .h de SDL sont dans /usr/include/SDL2.

Obtenir les options d'édition de liens (les -l) :

$ pkg-config --libs sdl2
Résultat : -lSDL2 — pour lier avec SDL, utilise -lSDL2.
En pratique

Au lieu de deviner les options, tu les injectes directement dans gcc :

$ gcc -o mon_jeu mon_jeu.c $(pkg-config --cflags sdl2) $(pkg-config --libs sdl2)

Le shell exécute pkg-config, récupère le résultat, et le passe à gcc. Tu n'as rien à retenir par cœur.

C'est juste un outil de confort pour ne pas avoir à chercher manuellement où sont installées les bibliothèques sur ton système.

Règles make
C'est quoi make ?

Tu en as marre de taper gcc -o cafeteria cafeteria.c -L. -lticket_tab à chaque fois ? Et de te rappeler quoi recompiler quand tu modifies un fichier ? make fait tout ça pour toi automatiquement.

La structure d'une règle
cible: source1 source2
    commande1
    commande2
  • cible → le fichier que tu veux produire
  • source1 source2 → les fichiers dont tu as besoin pour produire la cible (les dépendances)
  • commande1, commande2 → ce qu'il faut exécuter pour produire la cible
Comment make réfléchit ?

Make regarde la cible et se pose deux questions :

  • Est-ce que la cible existe ? Si non → il exécute les commandes.
  • Est-ce que la cible est plus ancienne qu'une de ses sources ? Si oui → il exécute les commandes, parce que ça veut dire qu'une source a été modifiée depuis la dernière fois.

Si la cible existe et qu'elle est plus récente que toutes ses sources, make ne fait rien. Pas besoin de recompiler.

Exemple concret
foo.o: foo.c
    gcc -c foo.c -o foo.o
  • Je veux produire foo.o
  • J'ai besoin de foo.c
  • Si foo.o n'existe pas ou si foo.c a été modifié après la création de foo.o → exécute gcc -c foo.c -o foo.o

Si tu ne touches pas à foo.c, make te dit "rien à faire". C'est l'intérêt : recompiler uniquement ce qui a changé.

Le piège classique
⚠ Les commandes doivent commencer par une tabulation, pas des espaces ! Si tu mets des espaces, make ne comprend pas et te donne une erreur bizarre. C'est un truc historique pénible mais c'est comme ça.
Makefile
C'est quoi make ?

Un programme qui compile automatiquement ton projet. Il sait quoi recompiler quand tu modifies un fichier, et il le fait tout seul.

Structure d'une règle
cible: source1 source2
    commande
  • Cible = ce que tu veux obtenir (ex : pacman, main.o)
  • Sources = ce dont tu as besoin pour le fabriquer (les dépendances)
  • Commande = ce qu'il faut exécuter (précédée d'une tabulation, pas d'espaces !)

Make exécute la commande si la cible n'existe pas ou si elle est plus ancienne qu'une de ses sources.

Exemple complet
makefile
main.o: main.c
    gcc -c main.c -o main.o

player.o: player.c
    gcc -c player.c -o player.o

pacman: main.o player.o
    gcc main.o player.o -o pacman

Si tu modifies juste player.c, make recompile seulement player.o puis pacman. Il ne touche pas à main.o.

Règle all
makefile
all: pacman

Par convention, all est la première règle du Makefile. Quand tu tapes juste make sans argument, make exécute la première règle. En mettant all: pacman en tête, tu t'assures que make tout court compile le projet entier.

Variables dans un Makefile
Pourquoi des variables ?

Imagine que tu écris gcc -Wall -Werror -std=c99 dans 10 règles différentes. Si tu veux changer une option, tu dois modifier 10 lignes. Avec une variable, tu changes une seule ligne.

Comment ça marche ?

Tu définis une variable :

makefile
CC = gcc
CFLAGS = -Wall -Werror -std=c99

Tu l'utilises avec $(NOM) :

makefile
main.o: main.c
    $(CC) $(CFLAGS) -c main.c -o main.o

Make remplace $(CC) par gcc et $(CFLAGS) par -Wall -Werror -std=c99 avant d'exécuter la commande.

Modifier une variable depuis le terminal

Tu peux écraser la valeur d'une variable quand tu lances make :

$ make foo CC=clang

Là, même si le Makefile dit CC = gcc, make utilisera clang à la place. La valeur donnée en argument écrase celle du Makefile.

= vs ?=
makefile
CC = gcc

Ça définit CC à gcc toujours, même si une variable d'environnement CC existe déjà.

makefile
CC ?= gcc

Le ?= veut dire "définis CC à gcc seulement si CC n'a pas déjà une valeur". Si tu as fait export CC=clang dans ton terminal avant, le Makefile garde clang.

En résumé :
NOM = VALEUR → définit toujours
NOM ?= VALEUR → définit seulement si pas déjà défini
$(NOM) → utilise la variable
make foo NOM=truc → écrase la variable depuis le terminal
Exemple complet — sans vs avec variables

Sans variables, ton Makefile ressemble à ça :

makefile
main.o: main.c
    gcc -Wall -Werror -std=c99 -c main.c -o main.o

player.o: player.c
    gcc -Wall -Werror -std=c99 -c player.c -o player.o

enemy.o: enemy.c
    gcc -Wall -Werror -std=c99 -c enemy.c -o enemy.o

pacman: main.o player.o enemy.o
    gcc main.o player.o enemy.o -o pacman

Tu vois gcc -Wall -Werror -std=c99 répété 3 fois. Tu veux ajouter -g pour le débogage ? Tu dois modifier 3 lignes.

Avec des variables :

makefile
CC = gcc
CFLAGS = -Wall -Werror -std=c99

main.o: main.c
    $(CC) $(CFLAGS) -c main.c -o main.o

player.o: player.c
    $(CC) $(CFLAGS) -c player.c -o player.o

enemy.o: enemy.c
    $(CC) $(CFLAGS) -c enemy.c -o enemy.o

pacman: main.o player.o enemy.o
    $(CC) main.o player.o enemy.o -o pacman

Tu veux ajouter -g ? Tu changes une seule ligne :

makefile
CFLAGS = -Wall -Werror -std=c99 -g
